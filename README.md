# HAIntly Main Frontend (`main-fe`)

Пользовательский интерфейс HAIntly на Next.js.

## Запуск

Скопируйте шаблон локального окружения и запустите приложение:

```bash
cp .env.example .env.local
npm install
npm run dev
```

Frontend доступен на `http://localhost:3100`. Шаблон задаёт `NEXT_PUBLIC_API_BASE_URL=http://localhost:8101`: этот адрес соответствует root orchestration, где Docker Compose публикует `main-be` с контейнерного порта `8000` на host-порт `8101`. Если `main-be` запускается отдельно без root orchestration и слушает host-порт `8000`, измените локальное значение на `http://localhost:8000`.

В окружении `main-be` origin `http://localhost:3100` должен быть разрешён CORS вместе с cookie credentials. `.env.local` игнорируется Git; `.env.example` не содержит секретов и отслеживается как шаблон.

Проверки: `npm run lint`, `npm test`, `npm run build`.

## Ответственность

- регистрация, подтверждение email и вход;
- подключение нескольких HH-аккаунтов через flow `main-be`;
- выбор резюме;
- запуск сбора вакансий и отображение его статуса;
- список вакансий, сортировка по AI score 1–100 и пользовательские статусы;
- генерация и редактирование сопроводительного письма согласно OpenSpec;
- получение уведомлений и обновлений по SSE.

## Интеграции

- HTTP ↔ `main-be`;
- SSE ← `main-be`.

Прямые обращения к другим сервисам, HH и AI-провайдеру запрещены.

## Архитектурные ограничения

- Pages/routes выполняют только композицию.
- UI не содержит прямых API-вызовов и бизнес-оркестрации.
- HTTP и SSE изолируются в типизированном API boundary.
- Loading, empty, error, success и background states проектируются явно.
- AI score и сгенерированный текст не отображаются как гарантированная истина.
- Резюме, пароли и другие персональные данные не логируются.
